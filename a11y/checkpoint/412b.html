Overview

In general, name, role, state, or value issues under this checkpoint relate to more complex widgets, especially ones that relate to existing ARIA authoring patterns such as tabpanels, dialogs, listboxes, sliders, carousels, etc.

For issues related to simpler components that have an HTML equivalent such as links, buttons, checkboxes, radio buttons, and selects, see the issues under checkpoint 4.1.2.a: Name, Role, Value.

For issues related to complex interactions that don’t have HTML or ARIA widget equivalents and are so broken that breaking them down into individual issues is not effective or productive, see checkpoint 4.1.3.c: Compatibility.

Name:

  • A label for a component
  • The name of a tab
  • The name of a dialog
  • The name of a carousel
  • The name of a slider

Role:

  • Tab
  • Dialog
  • Slider
  • Menu/menu item

Value or State:

  • Expanded/collapsed
  • Selected
  • Current
  • Disabled
  • Text in textfield
  • [x] of [x]
  • Slider’s value
  • etc.

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • The name and role of custom user interface components must be programmatically determinable by assistive technologies.
  • The states, properties, and values of custom user interface components which can be set by the user must be programmatically determinable by assistive technologies.
  • Notifications of changes to states, properties, and values of user interface components must be communicated to assistive technologies.
  • HTML markup must meet W3C specifications and facilitate accessibility support with assistive technologies.
  • ARIA should be used to indicate name, role, and value for tabs, modals, or other dynamic widgets.
  • The tabpanel (content section) of the tab should have an accessible name.
User Story: As a screen reader user, I want the name, role, and value of user interface components to be programmatically determinable so that I can get this information via my screen reader.
Success: If you were a screen reader user, when accessing user interface components, you would be able to determine the role, purpose, and state of the components.

Testing Logic

Is the name of the component read by a screen reader when the component receives focus?

  • Yes; PASS
  • No; See a href="/a11y/failure/412/f412ba.html">F4.1.2.b.a Component does not have an accessible name.

Web Test Procedure

  1. Using a screen reader, navigate to every user interface component in the page.
  2. Verify that the screen reader speaks a name for each component.
  3. Verify that the screen reader speaks a valid role for each component.
  4. Verify that the screen reader speaks any state, property, or value that can be set by the user.
  5. Verify that the screen reader notifies the user of any changes to states, properties, or values.
