Overview

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • Status messages must be conveyed to assistive technology users.
User Story: As a screen reader user, I want to hear status messages.
Success: If you were a screen reader user, when interacting with a site, you would hear any status messages.

Testing Logic

TEST 1: Is the content a status message?

Status messages include pop-ups which are not given focus, text indicating the number of search results, text informing that an action was completed, inline error or success messages, busy messages, progress bars, and modification or removal of existing status text. This includes non-text content, such as images.

  • Yes; Is the status message communicated by assistive technologies?
    • Yes; Is role="alert" or aria-live="assertive" used on content which is not critical or time-sensitive?
      • Yes; See
      • No; PASS
    • No; Does the status message communicate the success or results of an action or the state of an application?
      • Yes; See
      • No; Does the status message alert the user about an error, warning, or suggestion?
        • Yes; See
        • No; Does the status message communicate the progress of a process?
          • Yes; See
          • No;
  • No; Does the content use one of the status message roles or ARIA Live Regions to announce changes? (Examples include modal dialogs, expandable regions, and content such as additional form fields added after the current user focus.)
    • Yes; See 4.1.2.1.1 Button does not have an accessible name.
    • No; PASS

Web Test Procedure

Desktop

  1. Identify any status messages that appear on the page.
  2. Trigger each status message and verify that the newly added status message is announced by the screen reader.

Mobile

  1. Identify any status messages that appear on the page.
  2. Using VoiceOver or TalkBack, trigger each status message and verify that the newly added status message is announced by the screen reader.

Failures

  • 4.1.3.1.1 Snackbar/toast is not announced by the screen reader.
  • 4.1.3.1.2 Summary of content change is not announced by the screen reader.
  • 4.1.3.1.3 Feedback for an action is not announced by the screen reader.
  • 4.1.3.1.4 Error message is not announced by the screen reader.
  • 4.1.3.1.5 Progress is not announced by the screen reader.
  • 4.1.3.1.6 Busy message is not announced by the screen reader.
  • 4.1.3.1.7 Content update is not announced by the screen reader.
