Overview

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • Features and components must be compatible with assistive technologies.
  • Actions must not lead to only silence from the screen reader.
User Story: As a screen reader user, I want the name, role, and value of user interface components to be programmatically determinable so that I can get this information via my screen reader.
Success: If you were a screen reader user, when accessing user interface components, you would be able to determine the role, purpose, and state of the components.

Testing Logic

Is the name of the component read by a screen reader when the component receives focus?

  • Yes; PASS
  • No; See F4.1.2.b.a Component does not have an accessible name.

Web Test Procedure

  1. Using a screen reader, navigate to every user interface component in the page.
  2. Verify that the screen reader speaks a name for each component.
  3. Verify that the screen reader speaks a valid role for each component.
  4. Verify that the screen reader speaks any state, property, or value that can be set by the user.
  5. Verify that the screen reader notifies the user of any changes to states, properties, or values.

Failures

  • F4.1.2.c.a Custom user interface component is not compatible with assistive technology.
