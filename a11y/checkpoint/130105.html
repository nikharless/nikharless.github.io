Basic Requirements and Best Practices

= Requirement
= Best Practice

  • Lists must be marked up semantically as lists.
  • List markup must not be used on text which does not function as a list.
User Story: As a blind screen reader user, I want lists to be identified as such so that I can understand the structure of the content.
Success: If you could not see the web page, when you use a screen reader, you would know when you encountered a list.

Testing Logic

Are there any elements that function as lists on the page?

  • Yes; Are these elements marked up semantically as lists?
    • Yes; Does the list contain only allowed nested elements?
      • Yes; PASS
      • No; See {{ site.data.1301.1301[13010505].name }}
    • No; Is the list an unordered (bullet) list?
      • Yes; See {{ site.data.1301.1301[13010501].name }}
      • No; Is the list an ordered (numbered) list?
        • Yes; See {{ site.data.1301.1301[13010502].name }}
        • No; See {{ site.data.1301.1301[13010503].name }}
  • No; PASS

Are list tags used on text which is not a list purely for visual formatting purposes?

  • Yes; See {{ site.data.1301.1301[13010504].name }}
  • No; PASS

Web Test Procedure

Desktop

  1. Locate any text that looks and functions as a list.
    • List items may or may not have bullets or numbers.
    • List items may be presented vertically or horizontally.
  2. Verify that every element that functions as a list is identifiable as a list.
  3. Verify that the list type matches the type of list.
  4. Verify that content that does not function as a list is not marked up as a list.
    • If a list contains only one item, it is not a failure.

Mobile Screen Reader

VoiceOver or TalkBack

  1. Locate any text that looks and functions as a list.
  2. Verify that every element that functions as a list is identifiable as a list.
  3. Verify that content that does not function as a list is not spoken as a list.

Note: If bullets are removed from list items with CSS, VoiceOver does not read them as list items. This is not a failure if the list is marked up correctly.

Note: TalkBack speaks the number or "bullet" before each list item to identify lists.

Mobile App Test Procedure

iOS

This checkpoint is only applicable to lists that appear in webview content.

  1. Locate any text that looks and functions as a list.
  2. Verify that every element that functions as a list is identifiable as a list.
  3. Verify that content that does not function as a list is not spoken as a list.

Note: If bullets are removed from list items with CSS, VoiceOver does not read them as list items. This is not a failure if the list is marked up correctly.

Android

  1. Locate any text that looks and functions as a list.
  2. Verify that every element that functions as a list is identifiable as a list.
    • If the content is part of the native Android app:
      • Verify that one or both of the following is read:
        • "In List" / "Out of List"
        • "[number] of [number]" for each list item
    • If the content is part of a webview:
      • Verify that TalkBack speaks the number or "bullet" for each list item.
  3. Verify that content that does not function as a list is not spoken as a list.

PDF Test Procedure

PDF Accessibility Checker

This will identify incorrectly structured lists. You must do a manual check to verify that lists have been tagged as lists.

  1. Run PAC on the PDF.
  2. Check if Structure Tree is marked as a failure.
  3. If it is, activate the Results in Detail button.
  4. Under PDF/UA > Logical Structure > Structure Tree, check if "L" structure elements, "LI" structure elements, "Lbl" structure elements, or "LBody" structure elements are marked as failures.

Adobe Acrobat Pro

Accessibility Check

This will identify incorrectly structured lists. You must do a manual check to verify that lists have been tagged as lists.

  1. Select Tools > Accessibility > Accessibility Check.
  2. In the Accessibility Checker Options dialog, select Forms, Tables, and Lists from the Category drop-down.
  3. Make sure LI must be a child of L and Lbl and Body must be children of LI are checked.
  4. Activate the Start Checking button.
  5. In the Accessibility Checker results pane, check for failures under the Lists section.

Manual Check

  1. Locate all lists in the document:
    • Use the Reading Order tool:
      1. Select All tools > Prepare for accessibility > Fix reading order to open the Reading Order dialog.
      2. Make sure the Show page content groups checkbox is checked, and select the Structure types radio button. This will display the tag value of every tagged object in the Document pane.
      3. Uncheck Display like elements in a single block, so you can see the individual list tags.
      4. Scroll through the document to locate lists.
      OR
    • Use the Tags pane:
      1. Select the Tags button in the navigation pane.
      2. Select the first tag, and use the Up and Down Arrow keys to navigate through each element in the document.
  2. Verify that each element that functions as a list is tagged as a list.
    • Verify that the list is tagged as <L>.
    • Verify that each list item is tagged as <LI>.
    • If the list has bullets or numbers, make sure they are tagged as <Lbl>.
    • Verify that the list item content is tagged as <LBody>.
  3. Verify that the list elements are properly nested. 
    <L>
    <LI>
        <Lbl>
        <LBody>
    <LI>
    ...
  4. Verify that elements which are not lists are not tagged as lists.

Screen Reader

  1. Verify that every element on the page that functions like a list is spoken as a list by the screen reader.
  2. Verify that content that does not function as a list is not spoken as a list.
