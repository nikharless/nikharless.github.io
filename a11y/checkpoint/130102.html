Overview

Data tables are tables which require row or column headers to understand each cell's contents. If tables are not marked up correctly, assistive technology reads it as regular paragraph structure, stripping it of its meaning.

Layout tables are programmatic tables used solely for placement of content. If they are read by assistive technology as tables, the reading order will often not make sense.

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • Data tables must be marked up programmatically as tables.
  • Header cells must be programmatically associated with data cells.
    • If a table has both row and column headers, the scope attribute must be used.
    • Simple tables with only either row or column headers do not need the scope attribute.
    • If a table has more than one set of either row or column headers, the headers and id attributes must be used to associate them.
    • Empty header cells are not a failure.
  • Tables should not be used for layout purposes.
  • If a table is used for layout purposes, it must be marked up to prevent assistive technology from reading it as a data table:
    • Layout tables must not use <th> or <caption> elements.
    • Layout tables must not use headers, scope, or summary attributes.
    • Layout tables should use role="presentation" or role="none" to prevent assistive technology from trying to read it as a table.
  • Complex tables should be broken down into separate simple tables.
  • Tables should use the <caption> element to programmatically associate a visual label with the data table.
User Story: As a blind user, I want to have tables properly marked up so that my screen reader reads the data in a meaningful way.
Success: If you could not see the table, when you read the table with a screen reader, you could navigate from cell to cell and hear the header labels with the data to provide context.

Web Test Procedure

  1. Verify that data tables are marked up programmatically:
    1. Visually locate any data tables on the page, including images of data tables.
    2. Use ANDI: structures and select "reading order" to determine if the reading order makes sense, if needed.
    3. If ANDI identifies any tables, use ANDI: tables to view them.
    4. Use the "Analyze Next Table" button to view each table identified on the page.
    5. Verify that any visually apparent data tables are identified by ANDI. If they aren't, they are not marked up as tables.
    6. Verify that any data tables do not show role="presentation" in the Element information or under Accessibility Alerts.
    7. Review any Accessibility Alerts for role="table" coding errors.
  2. Verify that header cells are marked up programmatically:
    • Navigate to each data cell with ANDI:tables.
    • Verify that ANDI Output identifies all relevant row and column headers for each data cell.
      • If there are multiple levels of headers, verify that all are associated with the correct data cells.
  3. Verify that any captions are not marked up as headers.
  4. Verify that any layout tables are not identified as tables:
    1. Use ANDI: tables to locate any layouts which are identified as tables.
    2. Use the "Analyze Next Table" button to view each table identified on the page.
    3. Verify that the "Element" output does not show role="table".
    4. If the "Element" output does not show role="presentation", verify that ANDI Output and any alerts do not show header structure markup (e.g., <th>, scope="row").

Screen Reader Testing

NVDA

  1. Cycle through the tables using the T key.
  2. Use the up and down arrow keys to navigate through the cells of the table, or use Control+Alt+ up, down, left, and right arrow keys.
  3. Verify that header context is read with the data cells.

JAWS

  1. Cycle through the tables using the T key.
  2. Use Control+Alt+ up, down, left, and right arrow keys to navigate the table.
  3. Use Control+Alt+5 (on the numeric keypad) to read the row and/or column headers for the current cell.
  4. Verify that header context is read with the data cells.

VoiceOver

  1. Use VO+ up, down, left, and right arrow keys to navigate the table.
  2. Use VO+C to read the column header for the current cell.
  3. Verify that header context is read with the data cells.

Note: VoiceOver does not read row headers, though it will identify the row number.

Mobile App Test Procedure

  1. Locate any data tables on the page.
  2. Using the screen reader, navigate through the table and determine whether the content is understandable without being able to see it.
  3. If the table has interactive content, verify that it can be used with the screen reader turned on.

PDF Test Procedure

PDF Accessibility Checker

  1. Run PAC on the PDF.
  2. Select the WCAG tab and check if 1.3 Adaptable is marked as a failure.
  3. If it is, activate the Results in Detail button.
  4. Check if the following are marked as failures:
    • Table regularity
    • Table header cell assignments

Adobe Acrobat Pro

  1. Select Accessibility > Reading Order from the Task pane.
  2. Make sure the Show page content groups checkbox is checked, and select Structure types.
  3. Right-click on the table and select Table Editor from the context menu.
  4. Verify that header cells in data tables are marked up using <TH> tags.
  5. Right-click each header cell in the Document pane, and select Table Cell Properties from the context menu.
  6. In the Table Cell Properties dialog, verify that the appropriate Scope is set for the header cell.
  7. Verify that the values in the Row Span and Column Span attributes are accurate.

Screen Reader

  1. Press T to navigate through tables in the document.
  2. Use Ctrl+Alt+Up, Down, Right, or Left Arrow keys to navigate inside the table and away from table headers.
  3. Verify that table headers are announced along with the content of the data cells.
