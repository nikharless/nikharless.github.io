Overview

A component may gain focus with the keyboard (e.g., tabbing) or using the mouse (e.g., clicking). Hovering the mouse over a control does not move the focus (unless it is scripted to do so).

Clicking on buttons may also activate them, in which case a change of context is expected and is not a failure.

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • When a control gains focus, it must not result in a change of context.
    • When a component gains focus, focus must not be removed or redirected.
    • When a component gains focus, a new window, tab, or dialog must not be automatically launched.
    • When a component gains focus, there must not be a significant change of content before the focus.
    • When a component gains focus, a form must not be automatically submitted.
User Story: As a screen reader user, I want the context to not change unexpectedly so that I know where I am.
Success: If you were a screen reader user, when you navigate to a control, the context would not unexpectedly change.

Web Test Procedure

Desktop

  1. Using only the keyboard, navigate through all focusable components on the page.
  2. Verify that there is no unexpected change of context when any component gains focus.

Mobile

  1. Turn on the screen reader.
  2. Swipe to navigate through all focusable components on the page.
  3. Verify that there is no unexpected change of context when any component gains focus.
