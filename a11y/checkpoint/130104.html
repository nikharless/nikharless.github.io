Overview

The following are not failures of this checkpoint:

  • On pages with only one heading, that heading can be any level.
  • A level 1 heading is not required.
  • More than one Level 1 heading is allowed.
  • The level 1 heading isn't required to match the page title.
  • Headings do not have to be in sequence. For example, a sidebar may have a level 2 heading and appear in the DOM before the page's level 1 heading.
  • Skipping levels is not a failure as long as the proper hierarchy is maintained.

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • Visual headings must be marked up programmatically as headings.
  • Headings must be marked up hierarchically.
  • Heading markup must not be used on text which does not function as a heading.
  • Programmatic headings must not be empty.
  • The page should be divided up into sections with headings.
  • There should be a level 1 heading (<h1>) on every page.
  • There should only be one level 1 heading (<h1>) on each page.
User Story: As a blind screen reader user, I want headings to be marked up programmatically so that I can easily navigate the page and understand the structure of the content.
Success: If you could not see the web page, when you use a screen reader, you could navigate by headings and understand the structure of the page.

Testing Logic

Are there visible headings on the page?

  • Yes; Are the headings marked up programmatically as headings?
    • Yes; Are the headings marked up hierarchically?
      • Yes; PASS
      • No; See {{ site.data.1301.1301[13010402].name }}.
    • No; See {{ site.data.1301.1301[13010401].name }}.
  • No; This test is not applicable.

Are there programmatically identified headings that do not function as headings?

  • No; This test is not applicable.

Web Test Procedure

Desktop

  1. Use ANDI: structures to locate all programmatically identified headings.
  2. Verify that all visually apparent headings are identified as headings by ANDI.
  3. Compare the ANDI Output to the visual headings and verify that they match.
    • Note: If ANDI identifies a conflict between HTML markup and ARIA, the ARIA heading levels take precedence.
  4. Select the "view headings list" button in ANDI: structures to display the Structure Outline.
  5. Compare the heading levels listed in the Structure Outline to the page content and verify that the heading levels match the visual heading structure.
  6. Verify that content that does not function as a heading is not identified by ANDI as a heading.

Desktop Screen Reader

NVDA or JAWS

  1. Cycle through headings on the page using the H key (headings), OR use the 1 to 6 keys to find headings of the corresponding levels (if present), OR
    • NVDA: Use NVDA+F7 to prompt a dialog box showing all headings available on the page
    • JAWS: Use Insert+F6 to prompt a list of all headings
  2. Verify that every element that functions as a heading is spoken as a heading by the screen reader.
  3. Verify that content that does not function as a heading is not spoken as a heading.

VoiceOver

  1. Cycle through headings on the page using VO+Command+H (headings).
  2. Verify that every element that functions as a heading is spoken as a heading by the screen reader.
  3. Verify that content that does not function as a heading is not spoken as a heading.

Mobile Screen Reader

VoiceOver or TalkBack

  1. Locate any text that looks and functions as a heading.
  2. Verify that every element that functions as a heading is spoken as a heading.
  3. Verify that the heading levels properly convey the structural hierarchy of the content.
  4. Verify that content that does not function as a heading is not spoken as a heading.

Mobile App Test Procedure

iOS or Android

  1. Locate any text that looks and functions as a heading.
  2. Verify that every element that functions as a heading is spoken as a heading.
    • Heading levels are not announced in native app content, but they are announced in WebView content.
    • The screen title may or may not be read as a heading. If not, this is not a failure.
    • Modal titles may not be read as headings. This is not a failure.
  3. Verify that content that does not function as a heading is not spoken as a heading.

PDF Test Procedure

Adobe Acrobat Pro

  1. Select Tools > Accessibility > Reading Order to open the Reading Order dialog.
  2. Make sure the Show page content groups checkbox is checked, and select the Structure types radio button. This will display the tag value of every tagged object in the Document pane.
  3. Scroll through the document and locate all headings.
  4. Verify that each element that functions as a heading is tagged as a heading.
  5. Verify that the levels of the headings properly convey the structural hierarchy of the content.
  6. Verify that elements which are not headings are not tagged as headings.

Screen Reader

  1. Cycle through headings on the page using the H key (headings).
  2. Verify that every element on the page that functions like a heading is spoken as a heading by the screen reader.
  3. Verify that the heading levels properly convey the semantic hierarchy of the content.
  4. Verify that content that does not function as a heading is not spoken as a heading.
