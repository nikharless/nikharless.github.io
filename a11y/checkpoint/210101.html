Overview

This checkpoint applies to literal keyboards, but it also applies to any technology which uses the keyboard interface, such as switch controls, voice control, sip-and-puff controls, on-screen keyboards, and screen readers.

"Specific timings for individual keystrokes" refers to things such as multiple keystrokes within a short period of time or holding down a key for a length of time.

Examples of exceptions which depend on the path of the user's movement:

  • Free hand drawing or painting

Examples that are not path-dependent and must be made accessible:

  • Drag and drop
  • Resizing
  • Drawing a straight line
  • Drawing preset geometric shapes (circle, square, triangle, star, spiral)

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • All functionality must be available through the keyboard interface.
  • Keyboard functionality must not require specific timings for individual keystrokes.
  • Content should use the standard and expected keyboard interaction for the platform/user agent.
  • Instructions should be provided to explain any non-standard keyboard interaction required to use the functionality.
User Story: As a keyboard user, I want to be able to access all functionality with the keyboard so that I can use the page.
Success: If you used only a keyboard, when you access a page, you could use all functionality.

Web Test Procedure

Desktop Keyboard

  1. Using only the keyboard (no mouse or screen reader), navigate through the page.
    • Press the Tab key to navigate the focus through menu items, form inputs, links, help items, and other interactive UI elements.
    • Press Shift+Tab to navigate the focus in the reverse direction through all interactive UI elements.
    • In grouped inputs, menus, or list item elements, press the up, down, right, or left arrow keys to move between items inside the element.
    • Press the Enter key or the Space bar to select or activate the interactive UI element that has focus.
  2. Verify that all controls can be navigated to and activated using only the keyboard.

Desktop Screen Reader

  1. Turn on the screen reader.
  2. Follow the process for Desktop Keyboard.

Mobile

Screen Reader (iOS and Android)

  1. Turn on the screen reader.
  2. Swipe to each interactive element on the page.
  3. Verify that you are able to reach and use each interactive element.

Bluetooth Keyboard (Android)

  1. Pair a Bluetooth keyboard with your device.
  2. With TalkBack off, navigate through the screen using Tab and Arrow keys.
  3. Verify that you are able to reach and use all interactive elements.

Mobile App Test Procedure

Screen Reader (iOS and Android)

  • Follow the procedure for Web Mobile.

iOS

Voice Control

  1. Turn on Voice Control.
  2. Say “Show numbers.”
  3. Verify that a number appears next to all active (non-disabled) controls.
    • If you are unsure if a button is active or disabled, turn off Voice Control and try to activate the button with no assistive technology.
  4. If a number does not appear next to a link, turn off Voice Control and test with VoiceOver to ensure the link role is announced and the link can be activated with VoiceOver turned on.

Switch Control (optional)

  1. Using Switch Control, navigate through the app.
  2. Verify that you are able to reach and use all active (non-disabled) controls.

Keyboard (optional)

As of iOS 13, full keyboard support was enabled on iOS devices. To see the full range of keyboard commands for operating content, access the Help menu by pressing Tab+H.

Android

Switch Control

  1. Using Switch Control, navigate through the app. (See link under Additional Information.)
  2. Verify that you are able to reach and use all active (non-disabled) controls.

PDF Test Procedure
