Overview

In general, name, role, state, or value issues under this checkpoint relate to components that have an HTML equivalent, such as links, buttons, checkboxes, radio buttons, and selects.

For issues related to more complex widgets, especially ones that relate to existing ARIA authoring patterns such as tabpanels, dialogs, listboxes, sliders, carousels, etc., see the issues under checkpoint 4.1.2.b: Custom Controls.

For issues related to complex interactions that don’t have HTML or ARIA widget equivalents and are so broken that breaking them down into individual issues is not effective or productive, see checkpoint 4.1.2.c: Compatibility.

Name:

  • A label for a form control or button
  • The text of a link
  • The name of a tab
  • etc.

Role:

  • Button
  • Link
  • Text field
  • Radio Button
  • Checkbox
  • Select
  • Tab
  • Dialog
  • Slider
  • etc.

Value or State:

  • Expanded/collapsed
  • Selected
  • Current
  • Disabled
  • Text in textfield
  • [x] of [x]
  • Slider’s value
  • etc.

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • The name and role of user interface components must be programmatically determinable by assistive technologies.
  • The states, properties, and values of user interface components which can be set by the user must be programmatically determinable by assistive technologies.
  • Notifications of changes to states, properties, and values of user interface components must be communicated to assistive technologies.
  • An aria-label should be provided to describe sections with role="region".
  • Buttons (not links) should be used to allow users to activate scripted functionality (form submission, play/pause, etc.)
  • Links (not buttons) should be used to change focus or go to a new URL.
  • When multiple navigation regions are present on a page, aria-labels should be used to distinguish the purpose of each.
User Story: As a screen reader user, I want the name, role, and value of user interface components to be programmatically determinable so that I can get this information via my screen reader.
Success: If you were a screen reader user, when accessing user interface components, you would be able to determine the role, purpose, and state of the components.

Testing Logic

Is the name of the component read by a screen reader when the component receives focus?

  • Yes; PASS
  • No; Is the component a button?
    • Yes; See F4.1.2.a.a Button does not have an accessible name.
    • No; Is the component a text field, checkbox, radio button, or select?
      • Yes; See F4.1.2.a.b Form field does not have an accessible name.
      • No; Is the component an image button?
        • Yes; See F4.1.2.a.c Image button does not have an accessible name.
        • No; Is the component a link?
          • Yes; See F4.1.2.a.d Link does not have an accessible name.
          • No; Is the component an iframe?
            • Yes; See F4.1.2.a.e Iframe does not have an accessible name.
            • No; Is the component part of a group of components?
              • Yes; See F4.1.2.a.f Form element group does not have an accessible group name.
              • No;

Is the role of the component read by a screen reader?

  • Yes; Does the role match the function of the component?
    • Yes; PASS
    • No; Is the component a button?
      • Yes; See F4.1.2.a.g Button does not have a valid role.
      • No; Is the component a link?
        • Yes; Does the link have an href attribute?
          • Yes; See F4.1.2.a.i Link has an incorrect role.
          • No; See F4.1.2.a.j Link is missing the href attribute.
        • No; Is the component a select field?
          • Yes; See F4.1.2.a.k Select (combobox) has an incorrect role.
          • No; Is the component a select option?
            • Yes; See F4.1.2.a.l Select option is missing the option role.
            • No;
  • No; Is the component a button?
    • Yes; See F4.1.2.a.g Button does not have a valid role.
    • No; Is the component a text field?
      • Yes; See F4.1.2.a.h Form field has an incorrect role.
      • No; Is the component a link?
        • Yes; Does the link have an href attribute?
          • Yes; See F4.1.2.a.i Link has an incorrect role.
          • No; See F4.1.2.a.j Link is missing the href attribute.
        • No; Is the component a select field?
          • Yes; See F4.1.2.a.k Select component is missing required states or properties.
          • No; Is the component a select option?
            • Yes; See F4.1.2.a.l Select option is missing the option role.
            • No;

Does the component have a checked state (e.g., checkbox, radio button, or switch)?

  • Yes; Is the checked state read by the screen reader
    • Yes; Is the checked state spoken when changed?
      • Yes; PASS
      • No; See F4.1.2.a.w State change is not spoken by screen reader.
    • No; Is the component a checkbox?
      • Yes; See F4.1.2.a.o Checkbox is missing the checked state.
      • No; Is the component a radio button?
        • Yes; See F4.1.2.a.p Radio button is missing the checked state.
        • No; See F4.1.2.a.q Switch is missing the checked state.
  • No; Not applicable

Web Test Procedure

  1. Using a screen reader, navigate to every user interface component in the page.
  2. Verify that the screen reader speaks a name for each component.
  3. Verify that the screen reader speaks a valid role for each component.
  4. Verify that the screen reader speaks any state, property, or value that can be set by the user.
  5. Verify that the screen reader notifies the user of any changes to states, properties, or values.
