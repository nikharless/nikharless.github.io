Overview

This checkpoint covers the accessible name of components. The accessible name is the programmatic name identified and communicated by assistive technology. It usually should be the same as the visible label, but this isn't necessarily the case.

The following components must have accessible names:

  • Form components
  • Links
  • Other user interface controls such as menus, tabs, etc.
  • Widgets such as comboboxes, grids, trees, etc.
  • Dialogs
  • Frames and iframes

An accessible name may be provided by:

  • A programmatically associated <label> for a form control
  • An aria-label attribute
  • A title attribute
  • An aria-labelledby attribute
  • The text of a <button>
  • A value attribute for Submit and Reset buttons
  • An alt attribute for image buttons
  • The text of a link
  • The text of a drop-down option or tab
  • The title of a frame, or iframe
  • An aria-label or aria-labelledby element for a dialog
  • etc.

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • The name of user interface components must be programmatically determinable by assistive technologies.
  • The accessible name and description of user interface components must accurately describe the component's purpose.
  • If the name of the component changes during interaction, the name must be programmatically updated.
  • An aria-label should be provided to describe sections with role="region".
  • When multiple navigation regions are present on a page, aria-labels should be used to distinguish the purpose of each.
  • The tabpanel (content section) of the tab should have an accessible name.
User Story: As a screen reader user, I want the name of user interface components to be programmatically determinable so that I can get this information via my screen reader.
Success: If you were a screen reader user, when accessing user interface components, you would be able to determine the purpose of the components.

Web Test Procedure

Desktop

User Interface Components

ANDI
  1. Launch ANDI: focusable elements.
  2. Review each component and determine if the ANDI Output adequately describes the component.
  3. If the name of the component changes during interaction, repeat the above steps on the changed content.
Screen Reader
  1. Using a screen reader, navigate to every user interface component in the page.
  2. Verify that the screen reader speaks a name for each component.
  3. Verify that the name adequately describes the component.
  4. If the name of the component changes during interaction, repeat the above steps on the changed content.

Frames

Frames are obsolete in HTML5, but they may still be found on some outdated websites. Each frame's content must be tested with all other applicable tests as well.

  1. Launch ANDI. If a <frame> is used, ANDI will give an alert.
  2. Activate the Cancel button to test individual frames.
  3. Verify that each frame listed has a title.
  4. Select each link to view the frames.
  5. Verify that the title accurately describes the content of the frame.

Iframes

Each iframe's content must be tested with all other applicable tests as well.

  1. Use ANDI: iframes to identify any <iframe> on the page. If this option is not available, there are no iframes.
  2. Locate all iframes which have either no tabindex or a non-negative tabindex.
  3. Verify that the accessible name and description in the ANDI Output accurately describes the iframe content.

Mobile

  1. For iOS devices, turn off image recognition features.
  2. Follow the procedure for Desktop Web.

Mobile App Test Procedure

iOS

  1. Turn off image recognition features.
  2. Follow the procedure for Desktop Web.
  • Buttons: “[visible text or accessible name], button”
  • Links: “[visible text or accessible name], link”
  • Listview or tableview cells containing an interactive element: If no role such as “button” is spoken, use Voice Control to check for a number to appear when using the “Show numbers” command. VoiceOver is not required to announce a role such as “button” as long as a number appears with Voice Control.
  • Custom action menus: “Actions available”. If VoiceOver does not say “Actions available,” open the Rotor. Actions should become an available option in the Rotor, and the Rotor should automatically be set to Actions when it opens.
  • Page controls:
    • When focused on the page control, “Page [x] of [x], adjustable. Swipe up or down to adjust the value.” or similar.
    • After swiping up or down to switch pages: “Page [x] of [x]” and will continue to focus the page control.
  • Segmented controls: “[visible text or accessible name], button, [x] of [x]” or similar
  • Sliders: “[accessible name], [value of slider], adjustable. Swipe up or down to adjust the value.”
  • Steppers:
    • If buttons are separately focusable:
      • Decrement: “decrement, button” or similar
      • Increment: “increment, button” or similar
    • If buttons are focusable at the same time:
      • “[accessible name], [value], adjustable. Swipe up or down to adjust the value” or similar
  • Switches: “[accessible name], Switch button, off. Double tap to toggle switch” or similar
  • Text fields: “[accessible name], [entered value or placeholder text], text field, Double tap to edit. Actions available” or similar
  • Navigation bars: “[visible text only]”
  • Search bars: “[accessible name], [entered value or placeholder text], Search field, Double tap to edit” or similar
  • Tab bars: “[visible text or accessible name], Tab, [x] of [x]” or similar
  • Alerts: “Alert, [alert title]” or similar
  • Editable Text View: “[visible text or accessible name], Text field, Double tap to edit. Actions available” or similar
  • Selected: “[accessible name], button, selected”
    • Note: “button” may be optional, depending on whether the selected item is within a listview.
  • Expanded: “[accessible name], button, expanded [or] collapsed”
    • Note: “button may be optional, depending on whether the selected item is within a listview.
  • Disabled: “[accessible name], dimmed, button”

Android

  1. Follow the procedure for Desktop Web.
  • Buttons (including text, image, and floating action buttons):
    • “[accessible name], button”, OR
    • “[accessible name] (pause), Double tap to activate”, OR
    • “[accessible name], button (pause), Double tap to activate” or similar
  • Links:
    • An earcon plays identifying the text span as a link (“link” is NOT spoken), AND
    • “[accessible name], Double tap to activate” or similar
  • Alert or popup: “Alert” or “Popup”, then:
    • First text block, OR
    • Pane title as set by accessibilityPaneTitle
  • Badge: “[accessible name], has [x] notifications, [rest of announcement]”
  • Bottom navigation:
    • Selected state for currently active top-level destination, AND
    • “[accessible name], Double tap to activate” or similar
  • Checkbox or radio button:
    • “Checked” or “Not checked”, then “[accessible name], Double tap to activate” or similar
      • May or may not say “checkbox” or “radio button” before “Double tap to activate” or similar
  • Compound button or toggle button (i.e., a button with a “pressed” state):
    • “Checked” or “Not checked”, then [accessible name], Double tap to activate” or similar (no “checkbox” or “button” role is read)
  • Text field:
    • “[accessible name], Edit box”, OR
    • “[accessible name], Edit box for [accessible name]”, OR
    • “[entered value], Edit box for [accessible name]”, OR
    • “Editing Edit box for [accessible name]”, OR
    • “Editing [entered value], Edit box for [accessible name]”
      AND
    • “(pause) Double tap to enter text. Double tap and hold to long press”, OR
    • “(pause) Double tap and hold to long press” or similar
  • Expandable list view: “[accessible name], Collapsed (or) Expanded, Double tap to activate” or similar
  • Menu (various types): “[role], [accessible name of first menu item]
  • Progress bar:
    • Indeterminate: “Loading… [accessible name]”
    • Determinate: “Loading… [accessible name], [current %]”
  • Rating bar: “[Percentage], [accessible name], rating”
  • Seek bar:
    • “[Percentage], [accessible name], Seek control (pause), Use volume keys to adjust”, OR
    • “[Percentage], Seek control for [accessible name] (pause), Use volume keys to adjust” or similar
  • Snackbar or toast: A polite accessibility announcement of the text in the snackbar or toast
  • Spinner:
    • Collapsed state: “Drop-down list, [value] for [accessible name], Double tap to change, OR
    • “Drop-down list, [value], Double tap to change” or similar (may or may not say “drop-down list”
      AND/OR
    • Expanded state: “[accessible name], [x] of [x]” for each list item
  • Switch:
    • “[accessible name], [state], Switch (pause), Double tap to activate, OR
    • “[state], Switch for [accessible name] (pause), Double tap to toggle” or similar
