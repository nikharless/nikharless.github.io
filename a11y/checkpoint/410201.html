Overview

In general, name, role, state, or value issues under this checkpoint relate to components that have an HTML equivalent, such as links, buttons, checkboxes, radio buttons, and selects.

For issues related to more complex widgets, especially ones that relate to existing ARIA authoring patterns such as tabpanels, dialogs, listboxes, sliders, carousels, etc., see the issues under checkpoint 4.1.2.b: Custom Controls.

For issues related to complex interactions that don’t have HTML or ARIA widget equivalents and are so broken that breaking them down into individual issues is not effective or productive, see checkpoint 4.1.2.c: Compatibility.

Name:

  • A label for a form control or button
  • The text of a link
  • The name of a tab
  • etc.

Role:

  • Button
  • Link
  • Text field
  • Radio Button
  • Checkbox
  • Select
  • Tab
  • Dialog
  • Slider
  • etc.

Value or State:

  • Expanded/collapsed
  • Selected
  • Current
  • Disabled
  • Text in textfield
  • [x] of [x]
  • Slider’s value
  • etc.

Basic Requirements and Best Practices

= Requirement
= Best Practice

  • The name and role of user interface components must be programmatically determinable by assistive technologies.
  • The states, properties, and values of user interface components which can be set by the user must be programmatically determinable by assistive technologies.
  • Notifications of changes to states, properties, and values of user interface components must be communicated to assistive technologies.
  • An aria-label should be provided to describe sections with role="region".
  • Buttons (not links) should be used to allow users to activate scripted functionality (form submission, play/pause, etc.)
  • Links (not buttons) should be used to change focus or go to a new URL.
  • When multiple navigation regions are present on a page, aria-labels should be used to distinguish the purpose of each.
User Story: As a screen reader user, I want the name, role, and value of user interface components to be programmatically determinable so that I can get this information via my screen reader.
Success: If you were a screen reader user, when accessing user interface components, you would be able to determine the role, purpose, and state of the components.

Testing Logic

TEST 1: Is the name of the component spoken by a screen reader when the component receives focus?

  • Yes; PASS
  • No; Is the component a button?
    • Yes; See F4.1.2.a.a Button does not have an accessible name.
    • No; Is the component a text field, checkbox, radio button, or select?
      • Yes; See F4.1.2.a.b Form field does not have an accessible name.
      • No; Is the component an image button?
        • Yes; See F4.1.2.a.c Image button does not have an accessible name.
        • No; Is the component a link?
          • Yes; See F4.1.2.a.d Link does not have an accessible name.
          • No; Is the component an iframe?
            • Yes; See F4.1.2.a.e Iframe does not have an accessible name.
            • No; Is the component part of a group of components?
              • Yes; See F4.1.2.a.f Form element group does not have an accessible group name.
              • No;

TEST 2: Is the role of the component spoken by a screen reader?

  • Yes; Does the role match the function of the component?
    • Yes; PASS
    • No; Is the component a button?
      • Yes; See F4.1.2.a.g Button does not have a valid role.
      • No; Is the component a link?
        • Yes; Does the link have an href attribute?
          • Yes; See F4.1.2.a.i Link has an incorrect role.
          • No; See F4.1.2.a.j Link is missing the href attribute.
        • No; Is the component a select field?
          • Yes; See F4.1.2.a.k Select (combobox) has an incorrect role.
          • No; Is the component a select option?
            • Yes; See F4.1.2.a.l Select option is missing the option role.
            • No;
  • No; Is the component a button?
    • Yes; See F4.1.2.a.g Button does not have a valid role.
    • No; Is the component a text field?
      • Yes; See F4.1.2.a.h Form field has an incorrect role.
      • No; Is the component a link?
        • Yes; Does the link have an href attribute?
          • Yes; See F4.1.2.a.i Link has an incorrect role.
          • No; See F4.1.2.a.j Link is missing the href attribute.
        • No; Is the component a select field?
          • Yes; See F4.1.2.a.k Select component is missing required states or properties.
          • No; Is the component a select option?
            • Yes; See F4.1.2.a.l Select option is missing the option role.
            • No;

TEST 3: Does the component have a checked state (e.g., checkbox, radio button, or switch)?

  • Yes; Is the checked state spoken by the screen reader
    • Yes; Is the checked state spoken when changed?
      • Yes; PASS
      • No; See F4.1.2.a.z State change is not spoken by screen reader.
    • No; Is the component a checkbox?
      • Yes; See F4.1.2.a.p Checkbox is missing the checked state.
      • No; Is the component a radio button?
        • Yes; See F4.1.2.a.q Radio button is missing the checked state.
        • No; See F4.1.2.a.r Switch is missing the checked state.
  • No; Not applicable

TEST 4: Is the component a button?

  • Yes; Is the button a toggle?
    • Yes; Is the pressed state spoken by the screen reader?
      • Yes; PASS
      • No; See F4.1.2.a.m Toggle button is missing the pressed state.
    • No; Does the button control expandable content?
      • Yes; Is the expanded state spoken by the screen reader?
        • Yes; Is the expanded state spoken when changed?
          • Yes; Is the button a menu button?
            • Yes; Is the button identified as a menu button?
              • Yes; PASS
              • No; See F4.1.2.a.o Menu button is missing the aria-haspopup attribute.
            • No; PASS
          • No; See F4.1.2.a.z State change is not spoken by screen reader.
        • No; See F4.1.2.a.n Disclosure button is missing the expanded state.
      • No; Not applicable
  • No; Not applicable

TEST 5: Is the component currently disabled?

  • Yes; Does the screen reader announce the disabled state?
    • Yes; PASS
    • No; F4.1.2.a.v Disabled component is missing the disabled state.
  • No; Not applicable

TEST 6: Is the component required?

  • Yes; Does the screen reader announce the required state?
    • Yes; PASS
    • No; F4.1.2.a.w Required component is missing the required property.
  • No; Not applicable

TEST 7: Is the component read-only?

  • Yes; Does the screen reader announce the read-only state?
    • Yes; PASS
    • No; F4.1.2.a.x Read-only component is missing the read-only property.
  • No; Not applicable

Web Test Procedure

Desktop

  1. Using a screen reader, navigate to every user interface component in the page.
  2. Verify that the screen reader speaks a name for each component.
  3. Verify that the screen reader speaks a valid role for each component.
  4. Verify that the screen reader speaks any state, property, or value that can be set by the user.
  5. Verify that the screen reader notifies the user of any changes to states, properties, or values.

Mobile

  1. For iOS devices, turn off image recognition features.
  2. Follow the procedure for Desktop Web.

Note that default screen reader behavior may not include all information. For example, TalkBack does not read the currently selected value in <select> controls. These cases would fall out of the control of the developer and therefore are not considered failures.

Mobile App Test Procedure

iOS

  1. Turn off image recognition features.
  2. Follow the procedure for Desktop Web.

Note that iOS VoiceOver does not read roles for all elements (such as dialogs), so verify that the issue is, in fact, one that the developer likely has control over before filing a bug. See Accessibility Support for more information.

  • Buttons: “[visible text or accessible name], button”
  • Links: “[visible text or accessible name], link”
  • Listview or tableview cells containing an interactive element: If no role such as “button” is spoken, use Voice Control to check for a number to appear when using the “Show numbers” command. VoiceOver is not required to announce a role such as “button” as long as a number appears with Voice Control.
  • Custom action menus: “Actions available”. If VoiceOver does not say “Actions available,” open the Rotor. Actions should become an available option in the Rotor, and the Rotor should automatically be set to Actions when it opens.
  • Page controls:
    • When focused on the page control, “Page [x] of [x], adjustable. Swipe up or down to adjust the value.” or similar.
    • After swiping up or down to switch pages: “Page [x] of [x]” and will continue to focus the page control.
  • Segmented controls: “[visible text or accessible name], button, [x] of [x]” or similar
  • Sliders: “[accessible name], [value of slider], adjustable. Swipe up or down to adjust the value.”
  • Steppers:
    • If buttons are separately focusable:
      • Decrement: “decrement, button” or similar
      • Increment: “increment, button” or similar
    • If buttons are focusable at the same time:
      • “[accessible name], [value], adjustable. Swipe up or down to adjust the value” or similar
  • Switches: “[accessible name], Switch button, off. Double tap to toggle switch” or similar
  • Text fields: “[accessible name], [entered value or placeholder text], text field, Double tap to edit. Actions available” or similar
  • Navigation bars: “[visible text only]”
  • Search bars: “[accessible name], [entered value or placeholder text], Search field, Double tap to edit” or similar
  • Tab bars: “[visible text or accessible name], Tab, [x] of [x]” or similar
  • Alerts: “Alert, [alert title]” or similar
  • Editable Text View: “[visible text or accessible name], Text field, Double tap to edit. Actions available” or similar
  • Selected: “[accessible name], button, selected”
    • Note: “button” may be optional, depending on whether the selected item is within a listview.
  • Expanded: “[accessible name], button, expanded [or] collapsed”
    • Note: “button may be optional, depending on whether the selected item is within a listview.
  • Disabled: “[accessible name], dimmed, button”

Android

  1. Follow the procedure for Desktop Web.

Note that TalkBack does not read roles for all elements (such as dialogs), so verify that the issue is, in fact, one that the developer likely has control over before filing a bug. See Android accessibility: roles and TalkBack and Accessibility Support for more information.

  • Buttons (including text, image, and floating action buttons):
    • “[accessible name], button”, OR
    • “[accessible name] (pause), Double tap to activate”, OR
    • “[accessible name], button (pause), Double tap to activate” or similar
  • Links:
    • An earcon plays identifying the text span as a link (“link” is NOT spoken), AND
    • “[accessible name], Double tap to activate” or similar
  • Alert or popup: “Alert” or “Popup”, then:
    • First text block, OR
    • Pane title as set by accessibilityPaneTitle
  • Badge: “[accessible name], has [x] notifications, [rest of announcement]”
  • Bottom navigation:
    • Selected state for currently active top-level destination, AND
    • “[accessible name], Double tap to activate” or similar
  • Checkbox or radio button:
    • “Checked” or “Not checked”, then “[accessible name], Double tap to activate” or similar
      • May or may not say “checkbox” or “radio button” before “Double tap to activate” or similar
  • Compound button or toggle button (i.e., a button with a “pressed” state):
    • “Checked” or “Not checked”, then [accessible name], Double tap to activate” or similar (no “checkbox” or “button” role is read)
  • Text field:
    • “[accessible name], Edit box”, OR
    • “[accessible name], Edit box for [accessible name]”, OR
    • “[entered value], Edit box for [accessible name]”, OR
    • “Editing Edit box for [accessible name]”, OR
    • “Editing [entered value], Edit box for [accessible name]”
      AND
    • “(pause) Double tap to enter text. Double tap and hold to long press”, OR
    • “(pause) Double tap and hold to long press” or similar
  • Expandable list view: “[accessible name], Collapsed (or) Expanded, Double tap to activate” or similar
  • Menu (various types): “[role], [accessible name of first menu item]
  • Progress bar:
    • Indeterminate: “Loading… [accessible name]”
    • Determinate: “Loading… [accessible name], [current %]”
  • Rating bar: “[Percentage], [accessible name], rating”
  • Seek bar:
    • “[Percentage], [accessible name], Seek control (pause), Use volume keys to adjust”, OR
    • “[Percentage], Seek control for [accessible name] (pause), Use volume keys to adjust” or similar
  • Snackbar or toast: A polite accessibility announcement of the text in the snackbar or toast
  • Spinner:
    • Collapsed state: “Drop-down list, [value] for [accessible name], Double tap to change, OR
    • “Drop-down list, [value], Double tap to change” or similar (may or may not say “drop-down list”
      AND/OR
    • Expanded state: “[accessible name], [x] of [x]” for each list item
  • Switch:
    • “[accessible name], [state], Switch (pause), Double tap to activate, OR
    • “[state], Switch for [accessible name] (pause), Double tap to toggle” or similar
